# 2026-06-03 变更记录

## [fix][test] 修复事件反序列化时 extra_body 混入已声明字段的问题

### 变更内容
- 将 `src/event.rs` 中四类外层事件结构体改为手写 `Deserialize`，先反序列化完整事件对象，再依据已解析出的事件类型筛除已声明字段
- 保留 `extra_body` 对未知字段的兜底收集能力，同时避免 `time`、`self_id` 以及各类事件的判别字段、业务字段重复出现在 `extra_body`
- 新增消息事件与通知事件的回归测试，覆盖 `extra_body` 仅保留未知字段的预期行为

### 涉及文件
- `src/event.rs` [Changed] 重写事件外层反序列化逻辑并新增针对 `extra_body` 的单元测试
- `changelogs/2026-06-03.md` [Added] 记录本次事件反序列化修复内容

### 影响
- 调用方读取 `EventMessage`、`EventNotice`、`EventRequest`、`EventMetaEvent` 的 `extra_body` 时，将只看到协议中未声明的额外字段，不再混入已定义字段
- 现有通过结构体字段访问标准事件内容的用法不受影响，仅修正 `extra_body` 的内容边界

### 原因
- `serde(flatten)` 与内部标签枚举组合使用时，外层 `HashMap` 兜底字段收集会把已经参与事件分派和字段映射的键再次留下，导致 `extra_body` 语义失真
- 需要让 `extra_body` 更准确地承担“保留未知字段”职责，避免调用方重复判断或误用标准字段

## [refactor][BREAKING] 将消息段枚举改为匿名字段变体

### 变更内容
- 将 `ReceiveSegment` 与 `SendSegment` 的所有变体从 `Variant { data: ... }` 改为 `Variant(...)` 匿名字段形式
- 将两个枚举的 serde 标注调整为邻接标签形式 `#[serde(tag = "type", content = "data")]`，保持 OneBot 消息段在线上仍使用 `type + data` 结构
- 调整 `SendSegmentData::into_send_segment` 的全部构造逻辑，并修正 `src/event/message.rs` 中 `At` 消息段的直接构造写法
- 新增消息段序列化与反序列化回归测试，覆盖文本段、嵌套 `node` 段和空数据段

### 涉及文件
- `src/message/receive_segment.rs` [Changed] 将接收消息段枚举改为匿名字段变体并新增反序列化测试
- `src/message/send_segment.rs` [Changed] 将发送消息段枚举改为匿名字段变体并新增序列化测试
- `src/event/message.rs` [Changed] 同步更新 `QuickReplyAt` 中 `SendSegment::At` 的构造方式
- `changelogs/2026-06-03.md` [Changed] 追加记录本次消息段枚举重构

### 影响
- Rust 调用方若手写构造或模式匹配消息段枚举，需要从 `SendSegment::Text { data }` / `ReceiveSegment::Text { data }` 迁移到 `SendSegment::Text(data)` / `ReceiveSegment::Text(data)`
- OneBot 协议层的 JSON 结构保持不变，发送与接收仍使用 `{"type":"...","data":{...}}` 形式
- 启用 `selector` feature 时，`ReceiveSegment` 现在符合过程宏针对单匿名字段枚举的选择器生成形态

### 原因
- 匿名字段变体更贴合“枚举包装数据体”的语义，也让构造与匹配写法更简洁
- 需要在不改变协议线上数据结构的前提下，完成 Rust 层公开 API 的枚举写法重构

### 涉及文件
